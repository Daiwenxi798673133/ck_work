# 需求分析

> 项目：区域性别及人群流入流出离线分析（ClickHouse + Spring Boot + ECharts）
>
> 文档定位：本文锁定全系统的业务范围、功能边界与**统一口径**。其中第 5 章「术语定义」是核心，所有指标的计算口径以此为准；后续设计、实现、测试文档不得另起第二套定义。

---

## 1. 项目背景

电信运营商的信令网络会持续记录用户终端在各基站小区的驻留情况。把这些信令按时间切片聚合，就能反推出「某区域在某时段有多少人、人是从哪来的、又往哪去」，这类区域人口流动分析在城市治理、商圈选址、交通调度、应急疏散等场景里都有实际价值。

本项目做的是其中一个具体切面：**区域人群的流入、流出离线分析，以及流入流出人群的画像分布**。

几个关键定性：

- **离线（offline / batch）**：分析的是已经落库的历史数据，按批跑 SQL 出结果，不追求秒级实时。系统里没有 Kafka/Flink/Spark 这类流式组件，标题里的「离线」就是这个意思。
- **区域（region）**：分析粒度是「区域」，而非单个基站小区。区域由小区按 LAC（位置区码）天然聚合而来。
- **画像（portrait）**：在流入流出的基础上，进一步看这批流动人群的性别、年龄段、是否常住的分布。

数据沿用现有 `hbaseProcess` 工程的数据模型语义：上海市（行政区编码 310000）的 8 个基站小区，按 LAC 分成 3 个区域。原工程只是把这批小区的明细写进 HBase 做演示，本项目把它落到 ClickHouse 上做真正的流动分析。

## 2. 业务需求与功能需求

### 2.1 业务需求

用一句话概括业务目标：

> 给定一个区域和一个时间区间，能回答「这个区域的人是怎么流动的」——有多少人流入、多少人流出、多少人留下来了，这些流入流出的人在区域之间是怎么走的（OD），以及他们的性别、年龄、常住属性长什么样。

拆成可观测的业务问题：

- 某区域在某时段的在网人数、流入、流出、留存随时间怎么变化？
- 区域两两之间的人口流动量是多少（谁流向谁、流了多少）？
- 流入某区域的人群里，男女比例、各年龄段占比、常住与非常住占比分别是多少？流出的人群又如何？
- 演示和验证时，怎么一键把分析所需的基础数据准备好？

### 2.2 功能需求（5 大功能 = 5 个 API 端点）

系统对外提供 5 个功能，逐一对应一个 REST 端点。统一返回 `ApiResp{code, message, data}`，时间参数与返回均用 ISO 格式 `yyyy-MM-dd'T'HH:mm:ss`（时区 Asia/Shanghai）。

| 功能 | 说明 | API 端点 |
|---|---|---|
| **F1 区域查询** | 列出全部可分析区域及其基本信息（区域 id、中文名、LAC、小区数） | `GET /api/regions` |
| **F2 流入流出趋势** | 给定区域 + 粒度 + 起止时间，返回各时间窗口的在网人数 / 流入 / 流出 / 留存时序 | `GET /api/flow/trend?regionId=&granularity=hour\|day&start=&end=` |
| **F3 OD 区域间流动** | 给定粒度 + 起止时间，返回时段内区域两两之间的流动量矩阵（含对角线留存） | `GET /api/flow/od?granularity=&start=&end=` |
| **F4 流入流出画像分布** | 给定区域 + 粒度 + 时间 + 方向（流入/流出）+ 画像维度，返回该流动人群在该维度上的分桶分布 | `GET /api/flow/portrait?regionId=&granularity=&start=&end=&direction=in\|out&dimension=gender\|age_group\|is_resident` |
| **F5 数据初始化** | 一键灌入可复现的模拟数据，返回各表行数 + seed，用于演示与验证 | `POST /api/admin/init-data` |

各端点关键约束：

- **F1 区域查询**：无参，按区域 id 升序返回 3 条记录。
- **F2 流入流出趋势**：`granularity` 取 `hour` 或 `day`；结果按窗口起点升序，每个窗口包含 `population / inflow / outflow / retained` 四个指标。
- **F3 OD 区域间流动**：3 个区域两两组合最多 9 项（含 3 条对角线留存）；非对角线即跨区流动。
- **F4 流入流出画像分布**：`direction` 取 `in` 或 `out`；`dimension` 取 `gender`、`age_group`、`is_resident` 三选一；返回 `total` 与各分桶 `{bucket, count, ratio}`，且各桶 `count` 之和严格等于 `total`。
- **F5 数据初始化**：幂等，内部先清空事实表与派生表再重新生成；返回 seed、用户数与各表行数。

## 3. 非功能需求

| 类别 | 需求 | 落地方式 |
|---|---|---|
| **可复现性** | 同样的输入必须生成逐行一致的数据，分析结果可手算验证、可答辩 | 数据生成固定随机种子 `seed=20260601`；生成前先 `TRUNCATE` 事实/派生表，重复调用结果一致 |
| **离线批量** | 面向历史数据按批分析，不引入任何流式/实时组件 | 所有指标由 ClickHouse SQL 离线计算；无 Kafka/Flink/Spark |
| **口径一致性** | 全系统只允许一套流动口径，所有指标同源派生 | 趋势、OD、画像全部从同一份 OD 结果派生（见第 5 章） |
| **时区统一** | 时间窗口对齐不能因时区错位 | 容器、ClickHouse DDL（`DateTime('Asia/Shanghai')`）、JDBC 三处统一 `Asia/Shanghai` |
| **单机演示规模** | 单节点即可跑通，规模适中、便于在一台开发机上演示 | 单节点 ClickHouse（Docker）；数据规模见第 4 章 |
| **正确性可验证** | 守恒关系与黄金数据集可自动断言 | 26 个单元/集成测试全绿，含黄金数据集手算对照与守恒不变量断言 |

> 说明：本项目**不包含**实时分析、用户登录鉴权、权限控制、微服务拆分、数据 CRUD 管理界面等，这些不在作业范围内，详见第 5 章后的范围说明。

## 4. 数据需求

### 4.1 为什么用模拟数据

作业没有真实的电信信令数据可用，且真实信令涉及个人隐私，无法直接获取。因此本项目使用**程序生成的模拟数据**，在保留电信信令数据模型语义的前提下，构造一批可控、可复现的数据用于分析。

更重要的是：现有 `hbaseProcess` 工程里的模拟逻辑**无法直接用于流动分析**。原逻辑（`imitateDataSource.scala`）每个 5 分钟切片都用 `generateRandomHexes` 生成一批**全随机**的 IMSI，导致：

- 不同时间切片之间 IMSI 零重叠 → 留存恒为 0、流入流出恒为 100%，根本看不出「流动」；
- 每次运行结果都不同 → 不可复现，无法手算验证。

所以本项目重写了数据生成逻辑：用户身份在整个时间区间内**保持稳定**（同一用户跨切片可追踪），并固定随机种子保证可复现。

### 4.2 模拟假设

数据生成器（`DataGenerator`，由 F5 数据初始化接口调用）按以下假设构造数据：

- **用户规模**：10000 个用户。每个用户的 IMSI = `MD5("user_" + i)`（确定性，小写十六进制）。
- **时间范围**：`2026-05-31 00:00 ~ 2026-06-07 00:00`，5 分钟一个切片。其中 `05-31` 这一天是 **warm-up（预热）窗口**，目的是让正式上报起点 `2026-06-01` 的第一个时间窗口有真实的前驱窗口可比，避免边界缺前驱。
- **画像分布**（固定分布，可复现）：
  - 性别：男 55% / 女 45%。
  - 年龄：按 6 个年龄段加权抽样（权重 `0.05 / 0.15 / 0.25 / 0.25 / 0.20 / 0.10`），段内年龄均匀取整。
  - 常住属性：常住 60% / 非常住 40%。
  - 归属区域（home_region）：3 个区域等概率。
- **轨迹（关键）**：每个用户维护一个「当前所在区域」，初始等于其归属区域。每个 5 分钟切片：
  - 以 0.9 的概率判定「在线」。**离线则该切片不产生任何明细行**（这就是方案 A，详见第 5 章 transition-only）。
  - 在线时，以 0.85 的概率留在当前区域，否则等概率跳到另外两个区域之一；再在落点区域的小区列表里随机选 **1 个** cell。
  - 由此保证：每个 `(用户, 切片)` 至多产生一行、且恰好落在一个 cell（进而落在一个区域）。

### 4.3 与现有 hbaseProcess 模型的关系

区域与小区的口径完全沿用 `hbaseProcess` 的数据模型（来源：`imitateDataSource.scala:34-35`）：上海市（310000）共 8 个基站小区，按 LAC 天然聚合为 3 个区域。

| 区域 id | 区域名（示意） | LAC | 小区数 | 小区列表 |
|---|---|---|---|---|
| `310000_6254` | 陆家嘴 | 6254 | 3 | `310000_6254_140854179`、`310000_6254_140854180`、`310000_6254_140854181` |
| `310000_6234` | 人民广场 | 6234 | 2 | `310000_6234_140854182`、`310000_6234_140854183` |
| `310000_6200` | 徐家汇 | 6200 | 3 | `310000_6200_140854184`、`310000_6200_140854185`、`310000_6200_140854186` |

本项目对原模型做了两点必要扩展：

1. **新增画像维度**：原模型只有小区明细，没有性别/年龄/常住字段。本项目新增 `dim_user_profile` 维度表承载画像，让「画像分布」功能有数据可分。
2. **稳定可追踪的用户身份**：把原来「每切片全随机 IMSI」换成「稳定用户 + 固定 seed 的轨迹模型」，使跨窗口的留存与流动可计算、可复现。

### 4.4 数据规模（实测基线，seed=20260601）

| 表 | 含义 | 行数 |
|---|---|---|
| `dim_user_profile` | 用户画像维度 | 10,000 |
| `dwd_cell_imsi_5min` | 5 分钟基站明细（长表） | 18,144,589 |
| `dws_user_window_loc`（hour） | 小时级用户主导区域（派生） | 1,680,000 |
| `dws_user_window_loc`（day） | 天级用户主导区域（派生） | 70,000 |

明细约 1814 万行（≈ 10000 用户 × 2016 切片 × 在线概率 0.9），单机单节点即可承载，符合「单机演示规模」的定位。

## 5. 术语定义（核心 · 锁定口径）

> 本章是全系统的口径基准。所有指标只有一套定义，趋势、OD、画像三个功能的数值必须能相互对账。任何实现/测试/文档若与本章冲突，以本章为准。

### 5.1 区域（region）

按 LAC 对基站小区分组得到的分析单元。本系统共 3 个 region：`310000_6254`（陆家嘴）、`310000_6234`（人民广场）、`310000_6200`（徐家汇）。一个 region 下挂若干 cell（小区数分别为 3 / 2 / 3）。

### 5.2 主导区域 loc（loc(user, window) → region）

一个用户在一个时间窗口内可能出现在多个切片、多个小区。为了把「这个用户这个窗口属于哪个区域」唯一确定下来，定义：

> **loc = 该用户在该窗口内「最后一个 5 分钟切片」所在的 region。**

实现上等价于 `argMax(region_id, stat_time)`：取窗口内最大 `stat_time` 对应的 region。因为每个切片恰好落在一个 cell（进而一个 region），所以 loc 的结果唯一确定，不存在平票问题。`dws_user_window_loc` 表就是按此规则、对 hour 与 day 两种粒度各派生一份。

### 5.3 transition-only（方案 A · 离线不计入流动）

> **流动口径采用 transition-only：只统计「相邻两个窗口都在线」的用户。**

也就是说，OD 由相邻窗口的 loc 做 INNER JOIN 得到——只有当用户在前一个窗口和当前窗口都有 loc 记录时，这一次「从前窗 region 到当前窗 region」的移动才被计入。用户中途离线、上线、首次出现等情况，因为缺少相邻窗口的配对，**不计入任何流动**。

这条口径的直接后果是：用户离线再上线，不会被误算成「从最后已知位置移动过来」。例如某用户 W0 在 A、W1 缺席、W2 又在 A，系统不会把它算成 W0→W2 的留存或流动，因为 W1→W2 这一跳缺少 W1 的 loc。该行为在测试里有专门的黄金数据集断言守护。

> 局限性声明：transition-only 会忽略离线用户的流动，因此严格统计的是「连续在线人群」的流动。这是为换取口径自洽（守恒成立、可手算）而做的取舍，在「项目总结」文档中作为已知局限说明。

### 5.4 流入 inflow / 流出 outflow / 留存 retained

这三个指标连同在网人数，**全部从 OD 派生**，保证同源、可对账、人口守恒。对某区域 R、某窗口 T：

- **inflow（流入人次）** = `Σ_{R'≠R} OD(R'→R, T)`：从其他区域转移进入 R 的人次。
- **outflow（流出人次）** = `Σ_{R'≠R} OD(R→R', T)`：从 R 转移到其他区域的人次。
- **retained（留存人次）** = `OD(R→R, T)`：上一窗口在 R、当前窗口仍在 R 的人次（即 OD 矩阵的对角线）。

> 注意：以上均按「流动次数」计，即对 OD 行计数，不对用户去重。这样各画像分桶之和才能严格等于对应方向的流动总次数（见 5.7）。

### 5.5 在网人数 population

> **population = retained + inflow。**

即「上一窗口就在 R 且这窗口仍在 R 的人」加上「这窗口新流入 R 的人」，等于当前窗口在 R 的总在网人次。这是一个恒等式，作为守恒不变量在测试中逐窗口断言。

### 5.6 OD 矩阵（Origin-Destination）

区域两两之间的流动量矩阵。`OD(from→to, T)` = 上一窗口 loc 为 `from`、当前窗口 loc 为 `to` 的用户人次（相邻窗口 INNER JOIN 后按 from、to 分组计数）。

- 3 个区域 → 最多 3×3 = 9 项。
- **对角线（from == to）即 retained**（留存）；**非对角线即跨区流动**。
- 全局守恒：同一窗口内，`Σ_R inflow(R) == Σ_R outflow(R)`（所有跨区流入之和等于所有跨区流出之和）。

### 5.7 画像维度（portrait dimension）

对「流动人群」按画像维度做分桶分布。流动人群的取法与 OD 一致：

- **direction = in**：取 `from≠R 且 to=R` 的当前窗口用户（流入 R 的人）。
- **direction = out**：取 `from=R 且 to≠R` 的上一窗口用户（流出 R 的人）。

把这批用户 JOIN `dim_user_profile`，按指定维度分桶计数。**计数口径同样是「按流动次数计」**（对 OD 行计数，不对 IMSI 去重），因此某方向各桶 `count` 之和严格等于该方向的流动总次数，可与趋势的 inflow/outflow 对账。

三个画像维度：

| 维度 | 取值（分桶） |
|---|---|
| **gender（性别）** | `男`、`女`（底层 `Enum8('男'=1,'女'=2)`） |
| **age_group（年龄段）** | 固定 6 档：`<18`、`18-25`、`26-35`、`36-45`、`46-60`、`60+` |
| **is_resident（是否常住）** | `1`（常住）、`0`（非常住） |

### 5.8 age_group 边界

年龄段固定 6 档，边界规则如下（左闭右闭，按整数岁）：

| 年龄段 | 判定 | 边界示例 |
|---|---|---|
| `<18` | age < 18 | 17 → `<18` |
| `18-25` | 18 ≤ age ≤ 25 | 18 → `18-25`；25 → `18-25` |
| `26-35` | 26 ≤ age ≤ 35 | 26 → `26-35` |
| `36-45` | 36 ≤ age ≤ 45 | — |
| `46-60` | 46 ≤ age ≤ 60 | 60 → `46-60` |
| `60+` | age > 60 | 61 → `60+` |

### 5.9 ratio（占比）

`ratio = count / total`。当 `total == 0` 时 `ratio` 取 0（防除零、防 NaN）。各桶 ratio 之和约等于 1。

### 5.10 口径对账关系（汇总）

下面三组等式是全系统正确性的核心，测试里逐项断言：

1. **趋势 ↔ OD**：对区域 R 在区间内，`Σ trend.inflow(R)` == `Σ OD(to=R, from≠R) 的 flow`。
2. **人口守恒**：每个窗口 `population == retained + inflow`；全局 `Σ_R inflow == Σ_R outflow`。
3. **画像 ↔ 趋势**：某方向各画像桶 `count` 之和 == 该方向 `total` == 同区间同方向的流动总次数（in 对 inflow，out 对 outflow）。

## 6. 用例说明

### 用例 UC-1：查看某区域流入流出趋势

- **参与者**：分析人员
- **前置条件**：数据已初始化（已调用过 F5）
- **主流程**：
  1. 在看板选择区域（如陆家嘴 `310000_6254`）、粒度（hour）、起止时间（如 06-01 全天）。
  2. 系统调用 `GET /api/flow/trend`，返回各小时窗口的 population / inflow / outflow / retained。
  3. 看板用折线/面积图展示四条指标随时间的变化，概览卡片展示区间汇总。
- **结果**：分析人员看到该区域一天内人口流入流出的时序走势，且每个窗口满足 `population == retained + inflow`。

### 用例 UC-2：查看区域间 OD 流动

- **主流程**：选择粒度与时间区间 → 系统调用 `GET /api/flow/od` → 返回 ≤9 项 OD 流量 → 看板用桑基图（仅跨区链路）+ N×N 矩阵表（含对角线留存）展示。
- **结果**：看到区域两两之间的流动量，对角线为留存，可核对 `Σ(to=R, from≠R) == 趋势 inflow`。

### 用例 UC-3：查看流入/流出人群画像

- **主流程**：选定区域、时间、方向（in/out）与维度（gender/age_group/is_resident）→ 系统调用 `GET /api/flow/portrait` → 返回各桶 count 与 ratio → 看板用饼图（性别）、柱图（年龄段，流入 vs 流出对比）、环图（常住）展示。
- **结果**：看到流入或流出该区域人群的性别/年龄/常住分布，各桶之和等于该方向流动总次数。

### 用例 UC-4：初始化演示数据

- **主流程**：点击「初始化数据」→ 系统调用 `POST /api/admin/init-data` → 内部先清空事实/派生表，再用固定 seed 重新生成 → 返回 seed 与各表行数。
- **结果**：数据就绪且可复现（连续两次调用，各表行数完全一致）。
