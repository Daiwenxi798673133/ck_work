# 项目总结

> 项目：区域性别及人群流入流出离线分析（ClickHouse + Spring Boot + ECharts）
>
> 文档定位：回顾本项目的完成情况、踩过的坑和解决思路、还没做到的地方，以及如果继续往下做该怎么走。完成情况严格对照《需求分析》的功能清单与口径要求，难点和局限如实记录，不做拔高。

下面逐条对照《需求分析》里列出的需求，每一项都标注落地状态和验证依据。

### 1.1 五大功能 / 五个 API 端点

需求要求系统对外提供 5 个功能，各对应一个 REST 端点，统一返回 `ApiResp{code, message, data}`，时间用 ISO 格式（Asia/Shanghai）。全部完成：

|---|---|---|---|

| F2 流入流出趋势 | `GET /api/flow/trend` | 完成 | 24 个小时窗口，每窗口断言 `population == retained + inflow` |
| F3 OD 区域间流动 | `GET /api/flow/od` | 完成 | 3×3 共 9 项含对角线；跨区链路与 trend.inflow 对账一致 |
| F4 流入流出画像分布 | `GET /api/flow/portrait` | 完成 | gender/age_group/is_resident 三维度，各桶 `count` 之和严格等于 `total` |

- F4 的 `direction`（in/out）、`dimension`（gender/age_group/is_resident）按小写枚举契约解析，非法值返回 400（见 1.4 的小不一致说明）。
- F5 内部先 `TRUNCATE` 事实表与派生表再重新生成，返回 seed、用户数和各表行数。

### 1.2 四类前端图表

ECharts 单页看板，4 类图表全部对接真实接口数据并通过验收：

- **趋势折线/面积图**（`#chart-trend`）：在网人数 / 流入 / 流出 / 留存 4 条系列，切换粒度时数据点干净刷新（hour 24 点、day 1 点往返一致）。
- **OD 桑基图**（`#chart-od`）：左右二部图展示跨区链路，避开了多区互流的环（见 2.5）。
- **OD 矩阵表**（`#od-matrix`）：N×N 表格，对角线高亮为留存，非对角线按流量做热力底色。
- **画像分布图**（gender 饼图 / age_group 分组柱图 / is_resident 环图）：流入 vs 流出对比，占比直接用后端 `ratio`，前端不重算。

Playwright 只读验收：四类图表 canvas 均非空白，方向切换（in→out）只刷画像且数值随流向变化，全程 console error = 0。

## 2. 关键技术难点与解决

这一节是本项目最有价值的部分。下面 5 个难点，每个都先讲「问题是什么、为什么棘手」，再讲「怎么解决的」。

### 2.1 原 hbaseProcess 数据全随机，无法做流动分析也不可复现

**问题**：直接复用 `hbaseProcess` 的模拟逻辑（`imitateDataSource.scala`）会彻底卡死分析。原逻辑每个 5 分钟切片都用 `generateRandomHexes` 生成一批全新的随机 IMSI，结果是：

- 不同切片之间 IMSI 零重叠 → 同一个用户跨窗口根本追踪不到 → 留存恒为 0、流入流出恒为 100%，看不出任何「流动」。
- 每次运行结果都不一样 → 不可复现，没法手算验证、没法答辩。

这是个**根因级**问题，不是调参能绕过去的，必须重写数据生成。

**解决**：重构为「稳定用户 + 固定 seed + 轨迹模型」：

- 10000 个用户，IMSI = `MD5("user_" + i)`（确定性，跨切片可追踪）。
- 固定 `seed=20260601`，画像分布（性别 55/45、年龄 6 档加权、常住 60/40、归属区域等概率）全部由该 seed 派生，逐行可复现。
- 每个用户维护一个「当前所在区域」，每个切片以 0.9 概率在线、在线时 0.85 留守 / 否则等概率跳到另两个区域之一，再在落点区域随机选 1 个 cell。保证每个 `(用户, 切片)` 至多一行、恰好落在一个区域。

这一改让「同一用户跨窗口的位置变化」变成可计算的轨迹，留存和流动才有了真实分布；固定 seed 让结果可手算、可答辩。

### 2.2 流入流出与 OD 两套口径会对不上账

**问题**：流入、流出、留存、在网人数和 OD 矩阵，如果各算各的（比如 inflow 用一套 SQL、OD 用另一套），数值之间几乎必然对不上账。一旦看板上「趋势图的流入总和」和「OD 矩阵某列的跨区之和」对不齐，整个系统的可信度就崩了，而且这种 bug 极难定位。

**解决**：定下**单一口径、全部从 OD 派生**的铁律：

- 先用「相邻窗口 loc 做 INNER JOIN 再按 from/to 分组计数」算出唯一的 OD。
- inflow / outflow / retained / population / 画像全部从这份 OD 派生：
  - `inflow(R) = Σ_{R'≠R} OD(R'→R)`
  - `outflow(R) = Σ_{R'≠R} OD(R→R')`
  - `retained(R) = OD(R→R)`
  - `population(R) = retained + inflow`
- 配套统一 `loc(user, window)` 定义（窗口内最后一个切片所在 region，等价 `argMax(region_id, stat_time)`），因为每切片单 cell，loc 唯一确定、不存在平票。

同源派生让守恒可证：`population == retained + inflow` 是恒等式，全局 `Σ_R inflow == Σ_R outflow` 自然成立。这两条连同「画像桶和 == 方向 total」组成三向对账，测试里逐项断言。

### 2.3 可复现性：同样输入必须逐行一致

**问题**：「可复现」是答辩和自动化测试的前提。如果每次灌数结果不同，黄金数据集就无从手算，守恒断言也会时绿时红。

**解决**：三管齐下：

- **固定 seed**：`seed=20260601`，所有随机分布（画像 + 轨迹）从该 seed 派生，逐行确定。
- **幂等 init**：`DataGenerator.generate()` 开头先 `TRUNCATE` 事实表 / 派生表 / 画像表再重新生成；F5 接口直接复用它。实测连续两次调用 `dwd_cell_imsi_5min` 均为 18,144,589 行，各指标逐项一致。
- **黄金数据集手算对照**：`GoldenDatasetTest` 在隔离库 `flow_test` 灌入 5 个用户 ×2 区 ×3 窗口的小样本，手算出 OD@W1、OD@W2、trend、portrait 的期望值，再跑真实生产 SQL 断言，确保口径实现与手算一致。

### 2.4 时区 off-by-one：三处不统一就会窗口错位

**问题**：时间窗口对齐对时区极其敏感。容器、ClickHouse、JDBC 三层只要有一处时区不一致，`toStartOfHour` / `toStartOfDay` 的窗口边界就会偏移一小时甚至一天，导致相邻窗口配对错乱、对账全盘失败。这是典型的 off-by-one 陷阱，错了还不容易一眼看出来。

**解决**：三处强制统一 `Asia/Shanghai`：

- **docker**：容器 `TZ=Asia/Shanghai`。
- **DDL**：所有时间列用 `DateTime('Asia/Shanghai')`。
- **JDBC**：连接参数 `use_time_zone=Asia/Shanghai`。

这样生成端的 wall-clock、存储端的窗口切分、查询端的时间解析三方对齐，实测 `stat_time` 的 min/max 落在 `05-31 00:00 ~ 06-06 23:55`，与生成范围吻合，没有偏移。

### 2.5 ClickHouse 多区互流是有环图，标准桑基会报循环错

**问题**：3 个区域两两互流（A↔B、B↔C、A↔C 都有流量），从图论上看就是个**有环有向图**。ECharts 的 sankey 默认要求无环（DAG），直接把 OD 喂进去会触发循环检测报错，图画不出来。

**解决**：用**左右双列二部图**结构化避环：

- 起点节点名加前缀 `src\x01<region>`（depth 0，标签右对齐），终点节点名加前缀 `dst\x01<region>`（depth 1，标签左对齐）。
- 所有 link 一律 `source = src\x01from → target = dst\x01to`。同一个区域被拆成「左侧的它」和「右侧的它」两个节点，结构上不可能再成环。
- 显示时用正则去掉 `src/dst` 前缀，还原成纯区域名。

9 条 OD 过滤掉 3 条对角线（留存归矩阵表）后剩 6 条跨区链路、6 个节点（3 src + 3 dst），桑基图正常渲染，对角线留存则交给 N×N 矩阵表高亮展示。

## 3. 局限性（如实声明）

下面这些是本项目**没有做到或刻意取舍**的地方，如实列出，不回避。

1. **数据是模拟的，不是真实信令**。作业没有真实电信信令可用（且涉及隐私），全部数据由 `DataGenerator` 按固定假设生成。分析方法和口径是真的，数据分布是构造的，不能据此得出关于上海真实人口流动的结论。

2. **transition-only（方案 A）忽略离线用户的流动**。流动口径只统计「相邻两个窗口都在线」的用户，离线、上线、首次出现的用户不计入任何流动。例如用户 W0 在 A、W1 缺席、W2 又在 A，系统不会算成 W0→W2 的移动。好处是口径自洽、守恒成立、可手算；代价是严格统计的只是「连续在线人群」的流动，会系统性地低估真实流动量。这是为换取口径正确性而做的取舍。

3. **时间范围与规模有限**。数据只覆盖 7 天（含 05-31 预热窗口），10000 个用户，跑在单机单节点 ClickHouse 上。明细约 1814 万行，适合演示和手算验证，但不代表生产级数据量下的性能表现。

4. **OdController 的参数校验不一致**。如 1.4 所述，非法参数时 OdController 返回 HTTP 200 + code 400，与其它端点的真 400 不一致。功能上不影响判断成败，但 REST 契约不够干净。

5. **前端是单页，没有权限和分页**。ECharts 看板是一个静态单页，无登录鉴权、无权限控制、无数据分页或导出。区域只有 3 个、窗口数也有限，单页直出够用，但不具备多租户或大数据量场景下的可用性。

## 4. 收获心得

- **口径先行，再写代码**。本项目最关键的决定是「先锁死一套口径（统一 loc + transition-only + 全部从 OD 派生），再动手实现」。正是这一步让趋势、OD、画像三个功能天然对得上账，省掉了后期无穷无尽的「数值对不上」排查。需求文档第 5 章把口径锁成唯一基准，价值远大于任何一段实现代码。

- **可复现是测试的地基**。固定 seed 把「随机数据」变成「确定数据」，黄金数据集才有手算的可能，守恒断言才能稳定通过。没有可复现，自动化测试就是空中楼阁。

- **守恒不变量是最省力的正确性保险**。`population == retained + inflow` 和 `Σinflow == Σoutflow` 这类恒等式，写成断言后能一次性兜住一大类口径 bug，比逐个 case 验证高效得多。

- **跨层一致性问题（时区）要在源头统一**。时区这种横切关注点，与其在每个查询里临时修补，不如在 docker / DDL / JDBC 三处一次性统一，从根上消除 off-by-one。

- **可视化也有结构约束**。OD 是有环图、桑基要求无环，这种「数据结构和图表能力不匹配」的问题，用二部图重构比硬改图表库更干净。

## 5. 改进方向

如果继续往下做，按优先级大致是：

1. **接入真实信令数据**。把 `DataGenerator` 换成真实信令接入管道，验证口径在真实分布下是否依然自洽，并补上数据脱敏、隐私合规环节。

2. **实时管道（Kafka / Flink）**。当前是纯离线批处理。引入 Kafka 做信令摄入、Flink 做窗口聚合，可把分析从「按批跑」升级到「准实时」，支撑应急疏散这类对时效敏感的场景。

3. **扩规模与分布式 ClickHouse**。把用户数、区域数、时间跨度往上拉，单机会先遇到瓶颈。改用分布式表 + 多分片 ClickHouse，并对明细表做合理的分区与物化视图优化。

4. **方案 B：离线虚拟节点**。当前 transition-only 直接丢弃离线用户的流动。可引入「离线」虚拟节点，把上线/离线建模成「与离线节点之间的流动」，在保持守恒的前提下不再系统性低估流动量，作为方案 A 之外的可选口径。

5. **鉴权与多租户**。补上登录鉴权、基于角色的区域数据权限、多租户隔离，让看板从「演示单页」走向「可对外服务」。统一 OdController 的参数校验，让全部端点的 REST 契约一致。
